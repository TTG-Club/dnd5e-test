# D&D 5e (2024) — система для VTTG, собираемая вне приложения

Копия системы D&D 5e, вынесенная **из монорепы VTTG** и собираемая самостоятельно.
Задача репозитория — проверить доставку игровых систем в приложение по ссылке на
манифест и выявить, чего не хватает для полноценного SDK.

## Сборка

```bash
npm install
npm run build
```

Результат в `dist/` — готовая папка системы: `index.js` (сервер), `client.js` +
`client.css` (клиент), `system.json`, `data/` (справочники и SRD-компендиум).
Ровно эту папку CI пакует в архив релиза при пуше тега (см. [Релиз](#релиз)).

## Релиз

Версия системы — это **git-тег** вида `vX.Y.Z`, и только он. Вручную нигде номер
проставлять не нужно: на пуш тега GitHub Actions
([`.github/workflows/release.yml`](.github/workflows/release.yml)) сам штампует
версию в манифест, собирает систему и публикует релиз.

Как выпустить новую версию:

1. Влить изменения в `main`. Локально прогнать `npm run build` — те же проверки
   бандла гоняются в CI, и «негодный» бандл остановит релиз.
2. Выбрать следующий номер по SemVer: **patch** (`v0.0.X`) — фиксы, **minor**
   (`v0.X.0`) — новые возможности, **major** (`vX.0.0`) — несовместимые изменения.
   Последний тег: `git tag --sort=-v:refname | head -1`.
3. Создать аннотированный тег и запушить его вместе с веткой:

   ```bash
   git tag -a v0.0.4 -m "v0.0.4"
   git push origin main --follow-tags
   ```

Дальше всё делает CI:

- `scripts/stamp-manifest.mjs` проставляет в `system.json` `version` (из тега) и
  ссылки `url` / `download` / `manifest` / `readme` / `bugs` / … (из адреса
  репозитория). Поэтому значения этих полей в закоммиченном `system.json` — лишь
  заготовка: при релизе они перезаписываются.
- `npm run build` собирает `dist/` и прогоняет проверки бандла (см. «Проверка
  бандла»).
- содержимое `dist/` пакуется в `<id>.zip` (`system.json` — в корне архива).
- по тегу создаётся GitHub Release: к нему прикладываются архив и отдельный
  `system.json`. Канал обновлений — постоянная ссылка
  `releases/latest/download/system.json` (поле `manifest`): по ней VTTG видит
  новые версии.

Откатить ошибочный релиз: удалить тег локально и на origin
(`git tag -d vX.Y.Z && git push origin :refs/tags/vX.Y.Z`), затем удалить сам
Release на GitHub.

## Структура

| Папка | Что это |
|-------|---------|
| `src/engine` | движок правил D&D (переехал из `shared/src/system/dnd`) — **часть системы**, не SDK |
| `src/client` | интерфейс: листы, модалки, карточки, типы предметов, макросы |
| `src/server` | серверная точка входа системы |
| `sdk` | ⚠️ **временная копия** нейтрального ядра `@vtt/shared` — до публикации SDK-пакета |

## Манифест

Ключевые поля `system.json`: `manifest` (стабильная ссылка на свежий манифест — по
ней проверяются обновления), `download` (архив конкретной версии), `compatibility`
(минимальная и проверенная версия VTTG), `authors`, `url`, `license`, `readme`,
`bugs`, `changelog`, а также поля VTTG: `dataPath`, `client`, `defaultDistanceUnit`.

## Чего не хватает для полноценного SDK

Список получен эмпирически — по тому, что реально пришлось сделать руками, чтобы
система собралась вне монорепы.

1. **Нет опубликованного SDK-пакета.** Нейтральное ядро (`types`, `contracts`,
   `VttSystem`, `systemRegistry`, утилиты) скопировано в `sdk/`. Движок правил
   при этом переехал целиком — он принадлежит системе, а не платформе. Поверхность
   SDK, которую реально использует движок, узкая: `types/base`, `types/index`,
   `utils/{generateId,typeGuards,geometry,unitConverter}`, `system/contracts/*`,
   `system/vttSystem`.
2. **Не опубликован список модулей хоста.** 47 спецификаторов (`@/stores/…`,
   `@/shared_ui/…`, `@/core/…`, `pinia`, `@vueuse/core`, `@nuxt/ui/composables`),
   которые система резолвит в рантайме из `globalThis.__VTTHost`, перенесены
   вручную. Это главный публичный контракт платформы — он должен versioned-ся.
3. **Не опубликован пресет сборки.** Три обязательные настройки (`define`
   process.env, `interop: 'esModule'`, отключение авто-импорта компонентов Nuxt UI)
   воспроизведены вручную; при ошибке любая из них ломает систему неочевидно.
4. **Самоссылки через алиас приложения.** Исходники ссылались на собственные файлы
   как `@/systems/dnd5e/…` (46 мест) — снаружи такого пути нет, потребовался алиас.
5. **Ассеты приложения по абсолютным URL.** `/assets/modals/*.webp` отдаёт веб-корень
   VTTG; при внешней сборке их надо помечать внешними, иначе сборка падает.
6. **Идентификатор системы зашит в трёх местах:** `system.json`, `systemEntry.ts`
   (вызов `VTTSystems.register`) и движок (`readonly id`, `systemId`). Для системы,
   которую предлагается копировать и переименовывать, id должен браться из манифеста
   в одном месте.

## Проверка бандла

После сборки `dist/client.js` обязан пройти четыре проверки (иначе система молча
не поднимется в приложении):

- нет `process.env` — иначе `ReferenceError` до регистрации;
- есть вызов `VTTSystems.register`;
- нет своей копии `reka-ui` — иначе падает `setup` компонентов приложения;
- есть резолв `globalThis.__VTTHost`.
